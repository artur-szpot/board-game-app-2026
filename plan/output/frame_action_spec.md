# OPEN_FRAME / CLOSE_FRAME Specification

This document describes the expected frame navigation payloads and callback behavior for the app's stack-based window system.

## Frame model

Each open frame is represented by a stack entry:
- `id`: string (generated frame instance ID, e.g. cuid/UUID)
- `frame`: string (`SEARCH`, `OPTIONS`, `FORM`, or any custom frame key)
- `params`: object (payload for the opened frame, e.g. dataType, search options, form config)
- `callback?`: FrameCallback | undefined
- `state?`: object (optional local UI state such as partial form data or selection state)
- `isDirty?`: boolean (optional indicator that frame state has unsaved changes)

### Example frame entry

```json
{
  "id": "frame_123",
  "frame": "FORM",
  "params": {
    "screen": "ADD_GAME"
  },
  "callback": {
    "action": "OPEN_FRAME",
    "payload": {
      "frame": "VIEW_GAME",
      "id": "{returnedId}"
    }
  }
}
```

## OPEN_FRAME action

Payload shape:

```ts
interface OpenFramePayload {
  id?: string; // optional instance id; generated by state manager if omitted
  frame: string;
  params?: Record<string, any>;
  callback?: FrameCallback;
}
```

Behavior:
- push a new frame entry onto the stack
- preserve the old top frame as the callback owner
- do not immediately execute the callback
- if `id` is omitted, the UI/store generates a unique frame instance id

## CLOSE_FRAME action

Payload shape:

```ts
interface CloseFramePayload {
  id: string; // frame instance id that is closing
  result?: any; // returned frame data
  reason: 'confirm' | 'back' | 'close' | 'error';
}
```

Behavior:
- remove the frame with matching `id` from the stack
- if the closing frame was the top frame and its `reason` is `confirm`, deliver `result` to the saved callback of the previous frame
- if `reason` is `back`, `close`, or `error`, do not execute the saved callback
- if the closing frame is not the top frame, treat it as a cancellation and do not fire callbacks

## FrameCallback shape

```ts
interface FrameCallback {
  action: string;
  payload?: Record<string, any>;
}
```

Notes:
- the callback payload may include placeholder bindings such as `"id": "{returnedId}"` that are resolved from the closing frame result before dispatching
- for MVP, callbacks are primarily used to route the app flow after `CONFIRM`

## Navigation rules

- `BACK` closes the top frame with `reason: 'back'` and no result
- `CONFIRM` closes the top frame with `reason: 'confirm'` and a `result` object
- `SHUFFLE` is a frame-specific action and does not close the frame by default
- bottom frame is always `MAIN_MENU`
- top frame is the only frame rendered

## Example flow

1. `MAIN_MENU` opens `LIST_GAMES` with no callback
2. `LIST_GAMES` opens `VIEW_GAME` with a callback to return a selected game id
3. `VIEW_GAME` closes with `reason: 'confirm'` and `result: { id: 'game_123' }`
4. `LIST_GAMES` receives callback result and can dispatch `OPEN_FRAME` for the next screen

## MVP behavior notes

- no persistence of frame stack across page reloads
- no modal framing or overlay distinction; all frames are treated as full-screen screens
- if an API response arrives after the frame has been closed, the component should ignore it and not dispatch callback logic
- `BACK` does not expose a result and should not trigger the previous frame's callback
